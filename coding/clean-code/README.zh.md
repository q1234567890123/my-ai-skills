# Clean Code 中文版

路径：`coding/clean-code/`

## 这个技能是干什么的

这个技能用于让 AI 写出更容易维护、更容易测试、更容易修改的代码。它不追求“看起来优雅”，而是追求长期可维护。

适合场景：

- 代码命名混乱；
- 函数太长；
- 模块职责不清；
- 错误处理粗糙；
- 测试不好写；
- 需要重构但不想扩大风险；
- 想让 AI 做代码质量审查。

## 命名规则

- 类名和组件名表达“它是什么”。
- 函数名表达“它做什么”。
- 布尔变量使用 `is`、`has`、`can` 这类判断语义。
- 带单位的值必须写进变量名，比如 `timeoutMs`、`widthPx`。
- 避免 `data`、`info`、`helper`、`manager` 这种含糊名字。

## 函数规则

1. 一个函数只做一个层级的事。
2. 用卫语句减少嵌套。
3. 参数过多时改成结构化对象。
4. 不要在一个函数里同时做校验、转换、存储、渲染。
5. 函数名不能骗人：如果会发请求、读文件、写数据库，就不要叫 `getValue`。

## 模块规则

- 领域逻辑尽量不要依赖 UI、数据库、网络请求。
- 模块只暴露外部真正需要的东西。
- 不制造循环依赖。
- 副作用要放在明显边界。
- 公共 API 越窄，后续越好改。

## 错误处理

- 输入边界要校验。
- 错误信息要带上下文。
- 不要捕获错误后什么都不做。
- 不要过早把错误全部转成字符串。
- 如果要兜底，必须说明兜底规则。

## 重构流程

1. 先确认不能改变的行为。
2. 找到或补上测试。
3. 一次只做一种结构改动。
4. 每一步做验证。
5. 达到明显改善后停止，不追求完美。

## 质量检查

- 名字是否表达真实意图？
- 职责边界是否清楚？
- 是否有重复判断散落在多个地方？
- 错误是否可定位？
- 测试是否验证行为而不是私有实现？
- 这次改动是否保持最小范围？
